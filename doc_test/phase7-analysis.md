# 实验 7: 动态分区压测

> **测试日期**: 2026-06-07
> **环境**: 5 节点 YugabyteDB RF=3（Docker Compose + tc netem 30/60/90/120/150ms）

---

## 1. 实验设计

### 1.1 目的

在持续写入过程中注入网络分区（隔离 region2），观测读写行为的三阶段变化。

### 1.2 测试流程

| 时间 | 事件 |
|------|------|
| Phase 1 (0-10s) | 正常运行：后台持续写入 + 前端定时读取 |
| Phase 2 (10-25s) | 隔离 region2 (yb-2)：iptables 双向 DROP |
| Phase 3 (25-30s) | 恢复 region2：iptables 规则清除 |

---

## 2. 结果

### 2.1 Phase 1: 正常运行

| 读 # | 延迟 | 写入状态 |
|------|------|---------|
| 1 | 233ms | ✅ 成功 |
| 2 | 342ms | ✅ 成功 |
| 3-10 | 279-312ms | ✅ 成功 |

**读 avg ≈ 284ms, 写入全部成功**

### 2.2 Phase 2: 隔离 region2

| 读 # | 延迟 | 写入状态 |
|------|------|---------|
| 1 | **33,120ms** | ❌ 写入失败 |
| 2-15 | 56-98ms | ❌ 写入持续失败 |

**隔离后首次读取延迟 33s（Raft leader 切换），后续读取 avg ≈ 77ms，写入全部失败**

### 2.3 Phase 3: 恢复 region2

- 写入立即恢复（网络连通后 INSERT 成功）
- 读取延迟恢复正常

---

## 3. 分析

### 3.1 首次读取延迟 33s 的原因

隔离 region2 后，首次 SELECT 查询花了 **33.1 秒**，这是因为：

1. 客户端连接到 yb-2，但 yb-2 被隔离无法与其他节点通信
2. perf_test 表的 tablet leader 可能在 yb-2 上
3. 客户端尝试从 yb-2 读取 → tablet leader 不可达 → 等待 Raft leader 切换
4. leader 切换需要 heartbeat timeout + election timeout ≈ **数秒到数十秒**
5. 在延迟环境下（60ms egress on yb-2），leader detection 更慢

### 3.2 后续读取延迟反而更低 (56-98ms)

隔离后后续读取延迟反而从 284ms 降低到 **56-98ms**，原因：

1. 首次读取后，客户端知道 yb-2 不可用，转而连接其他节点（yb-1/yb-3 等）
2. 从 follower 读取：没有 leader 的锁竞争开销
3. follower 上的数据是过时的但仍然可读（默认 `yb_read_from_followers` 行为）

### 3.3 写入全部失败

写入（INSERT）在隔离期间全部失败：
- INSERT 需要 tablet leader 确认写入
- leader 切换后，新 leader 在其他节点上
- 但 yb-2 被隔离，从 yb-2 发出的请求无法到达其他节点
- **关键**: 即使集群已经恢复（新 leader 在 region1/3），从被隔离节点发出的请求仍然失败

### 3.4 恢复后立即生效

iptables 规则清除后：
- 网络立即恢复（无需等待 Raft 重新选举）
- 写入和读取立即成功
- **RPO = 0**：已提交的写入不会丢失

---

## 4. 结论

| 验证项 | 结果 |
|--------|------|
| Phase 1 正常读取 | ✅ avg ~284ms |
| Phase 2 首次读取 | ⚠️ 33s (leader 切换) |
| Phase 2 后续读取 | ✅ avg ~77ms (follower, 无锁竞争) |
| Phase 2 写入 | ❌ 全部失败 (隔离节点不可达) |
| Phase 3 恢复 | ✅ 立即恢复, RPO=0 |

**核心发现**: 网络分区期间，被隔离节点的首次查询会经历长达 33s 的 leader 切换延迟，但后续查询反而更快（follower 无锁竞争）。写入在隔离期间全部失败，恢复后立即生效。
