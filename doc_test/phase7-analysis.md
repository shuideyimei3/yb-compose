## 三次复现结果

恢复后的读取延迟：

| Run ID | 恢复后 5 次读取 |
|---|---|
| `20260608T082336Z` | 245ms, 214ms, 285ms, 297ms, 289ms |
| `20260608T085229Z` | 204ms, 246ms, 276ms, 500ms, 263ms |
| `20260608T093857Z` | 371ms, 534ms, 222ms, 713ms, 238ms |

脚本三次均给出一致结论：

- 隔离窗口内写入可能短暂失败，取决于 tablet leader 和客户端路径
- 健康节点上的读取大多仍可成功
- 首次故障路径可能出现长尾延迟
- 分区恢复后集群立即重新收敛
- 已提交事务不受分区影响，RPO=0

## 分析

隔离单个 region 时，RF=3 集群仍有多数派。只要客户端连接的是健康节点，整体服务通常不会完全不可用。长尾来自旧 leader 路由、连接超时、Raft 重新选举和请求重试。

Run 3 恢复后出现 713ms 读延迟，说明恢复并不等于所有路径立即回到基线；路由和 leader 缓存仍可能短时间内带来尾延迟。

## 结论

单 region 分区不会破坏 RF=3 集群的一致性。可用性表现取决于 leader 位置和客户端接入点。生产上应把客户端重试、超时和 leader placement 纳入故障恢复设计。
