## 关键观测

三次新测试均显示，时钟回退后的行为不能简单描述为“yb-5 SQL 服务不可用”。实际观测为：

- 时钟回退后，脚本检查到 `yb-5 postgres 仍可接受 SQL 连接`
- 分区 + 时钟异常组合下，通过 yb-5 写入失败，错误表现为连接被关闭，脚本判定为 `FAIL: no consensus`
- 最终集群状态仍可见 5 个节点

## 分析

YugabyteDB 使用 HLC 保证时间戳单调。物理时钟快进不会破坏单调性；物理时钟回退时，HLC 不应倒退。该机制与 Spanner TrueTime 的硬件保证不同：YugabyteDB 不依赖 GPS/原子钟，而是在软件层处理时钟异常。

本实验中的 -4s 回退没有导致 postgres 长期关闭，这说明不能把 HLC 安全机制简单表述为“关闭 SQL 服务”。更准确的结论是：HLC 维护单调性；当时钟异常叠加网络分区导致无法形成共识时，写入被拒绝，从而避免不一致提交。

## 结论

三次测试均支持：HLC 是 YugabyteDB 保证全局顺序的重要机制，时钟快进可安全处理，时钟回退不会让 HLC 倒退；在分区和时钟异常叠加时，系统通过拒绝无共识写入保证安全性。
