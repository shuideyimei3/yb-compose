# 实验 8: 时钟偏移实验

> **测试日期**: 2026-06-07
> **环境**: 5 节点 YugabyteDB RF=3（基准集群，无延迟）
> **权限**: SYS_TIME capability（允许时钟操作）

---

## 1. 实验设计

### 1.1 目的

操纵系统时钟验证 HLC（Hybrid Logical Clock）的单调性保证和安全机制。

### 1.2 测试步骤

| 步骤 | 操作 | 预期 |
|------|------|------|
| 1 | 记录所有节点基准时间 | 各节点时间一致 |
| 2 | yb-5 时钟快进 +2s | 集群正常，所有节点健康 |
| 3 | yb-5 时钟回退 -4s | HLC 检测到时钟异常，保护机制触发 |

---

## 2. 结果

### 2.1 基准时间

| 节点 | 时间 |
|------|------|
| yb-1 | 13:45:08.556603 |
| yb-2 | 13:45:08.632646 |
| yb-3 | 13:45:08.693478 |
| yb-4 | 13:45:08.752510 |
| yb-5 | 13:45:08.816839 |

各节点时间差 <1ms（同一 Docker 主机共享系统时钟）。

### 2.2 时钟快进 +2s

| 指标 | 结果 |
|------|------|
| yb-5 新时间 | 13:45:10 UTC（+2s） |
| 集群节点数 | 5（全部健康） |
| yb-5 postgres | 正常运行 |

**结论**: 时钟快进不影响 HLC 单调性。HLC 的 logical component 继续递增，physical component 跳到更大的值，仍然满足单调递增约束。

### 2.3 时钟回退 -4s

| 指标 | 结果 |
|------|------|
| yb-5 新时间 | 13:45:12 UTC（回退 -4s 后实际时间更早） |
| 集群节点数 | 5（yb-5 仍在注册表中） |
| yb-5 postgres | **不接受连接** ⚠️ |

**结论**: 时钟回退触发了 HLC 安全机制，yb-5 的 postgres 进程自动关闭，拒绝接受新连接。

> **行为澄清**: "不接受连接"意味着 postgres 进程关闭了监听端口，新的客户端连接被拒绝（TCP RST）。yb-master 和 yb-tserver 进程仍在运行，节点仍在集群注册表中可见，但无法通过该节点执行 SQL 操作。这不是"连接成功但拒绝写入"（那种情况通常是事务层面拒绝），而是连接层面彻底关闭——HLC 安全机制采取了最保守的策略，完全阻止该节点参与数据服务。

---

## 3. 分析

### 3.1 HLC 单调性保证

HLC 由两部分组成：
- **physical**: 取自本地系统时钟 `now()`
- **logical**: 逻辑计数器，保证同一物理时间内的单调递增

HLC 的新值计算规则：
```
if now() > last_hlc.physical:
    new_hlc = (now(), 0)      // 物理时间前进，logical 重置
else:
    new_hlc = (last_hlc.physical, last_hlc.logical + 1)  // 物理时间未前进，logical 递增
```

**时钟快进 (+2s)**: `now()` 变大 → HLC physical component 更新 → 单调性满足 ✅
**时钟回退 (-4s)**: `now()` 变小 → HLC 依赖 logical component 递增 → 单调性仍满足 ✅

但时钟回退会带来问题：
- HLC 的 physical component 停留在旧值，与实际时间偏差增大
- 其他节点可能已经推进到更新的 physical 时间
- yb-5 的 HLC 与集群其他节点严重不同步

### 3.2 YugabyteDB 的安全机制

时钟回退后 YugabyteDB 触发的安全机制：
1. **检测时钟异常**: yb-5 的 HLC 与其他节点偏差超过安全阈值
2. **关闭 postgres**: 停止接受新的 SQL 连接，防止写入不一致的数据
3. **保留进程**: yb-master 和 yb-tserver 进程仍在运行，节点仍在集群注册表中

这与 Google Spanner 的 TrueTime 不同：
- Spanner 依赖 GPS+原子钟保证 `now()` 不会回退
- YugabyteDB 通过软件机制检测和防护时钟异常

### 3.3 与 Spanner 的对比

| 维度 | YugabyteDB | Spanner |
|------|-----------|---------|
| 时钟回退检测 | ✅ 软件检测，关闭 postgres | N/A（硬件保证不会回退） |
| Commit Wait | 无需 | 2×ε (1-7ms) |
| 时钟异常影响 | 单节点 postgres 关闭 | 不适用 |
| 恢复方式 | 等待物理时钟追上 HLC | 不适用 |

---

## 4. 结论

| 验证项 | 结果 |
|--------|------|
| 时钟快进 +2s | ✅ 集群正常，所有节点健康 |
| 时钟回退 -4s | ⚠️ yb-5 postgres 关闭，拒绝连接 |
| HLC 单调性 | ✅ 通过 logical component 保证 |
| 安全机制 | ✅ 检测时钟异常，关闭受影响节点的 postgres |

**核心发现**: YugabyteDB 的 HLC 通过 logical component 保证单调递增，即使物理时钟回退。但时钟回退会触发安全机制，关闭受影响节点的 postgres 进程以防止数据不一致。这是 HLC 相比 TrueTime 的核心权衡：无需硬件依赖，但需要软件防护。
