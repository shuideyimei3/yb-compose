### 3.1 延迟对比

#### 基准环境（NET_DELAY_MS=1，即无额外延迟）

| 场景 | 读平均(ms) | 读P50(ms) | 读P99(ms) | 写平均(ms) | 写P50(ms) | 写P99(ms) |

#### 延迟环境（NET_DELAY_MS=30，各 zone 延迟=30×zone编号）

tc qdisc 配置：
- region1: `netem delay 30ms`
- region2: `netem delay 60ms`
- region3: `netem delay 90ms`

| 连接方式 | 节点延迟 | 读平均(ms) | 读P50(ms) | 读P99(ms) | 写平均(ms) | 写P50(ms) |
|---------|---------|-----------|----------|----------|-----------|----------|
| 从外部容器→region1 | 30ms egress | 93.3 | 89.5 | 151.7 | 91.1 | 89.8 |
| 从外部容器→region2 | 60ms egress | 127.9 | 126.4 | 186.4 | 120.7 | 120.6 |
| 从外部容器→region3 | 90ms egress | 168.7 | 166.4 | 247.3 | 177.8 | 170.8 |

**结论**：延迟差异与实际 egress 延迟增量成线性关系（30ms 增量 → ~35-40ms 延迟差），tc netem 成功模拟跨 region 网络延迟。基准测试中的 ~60ms 固定开销来自 Docker 网络栈 + psql 连接建立。

## Phase 4: 进阶实验

### 4.1 Commit Wait 分析
- 单主机环境下本地事务与跨 region 事务提交耗时相近
- YugabyteDB 使用 HLC 无需显式 Commit Wait（对比 Spanner 的 TrueTime 需 1-7ms）

### 4.2 正确性验证

| 测试 | 结果 |
|------|------|
| 读后写一致性 | ✓ 通过 |
| 并发转账（余额守恒） | ✓ 通过，初始 1000 → 最终 1000 |
| 冲突事务 | ✓ Read Committed 正确处理 |

### 4.3 故障切换对比

| 场景 | RTO (ms) | RPO | 方式 | 说明 |
|------|---------|-----|------|------|
| 停止 region1 节点 | ~400 | 0 rows lost | `docker stop` | 进程级故障，Raft 立即感知 |
| **iptables 隔离 region1** | **~987** | **0 rows lost** | **网络分区** | **真实网络分区模拟，需等待 Raft heartbeat timeout (~500ms)** |

> **关键发现**: iptables 网络分区的 RTO 约为 docker stop 的 2.5 倍。
> - `docker stop`: 进程死亡 → Raft 通过 TCP 断连立即感知 → 快速 leader 选举
> - `iptables isolate`: 进程存活但网络不通 → 需等待 Raft heartbeat timeout (默认 ~500ms) → 发起 leader 选举 → 额外等待
> - 在真实生产环境中，网络分区比进程崩溃更常见，iptables 模拟更贴近实际

### 4.4 分区下的一致性验证

| 场景 | 结果 | 现象 |
|------|------|------|
| region1 隔离期间从 region2/3 读写 | ✓ 一致 | 3 rows 完全一致 (pre-partition + during-partition) |
| 从隔离节点 (region1) 读取 | ✓ 自动保护 | postgres FATAL: database system is shutting down |
| 恢复后全节点数据收敛 | ✓ 一致 | 3/3 节点数据完全相同 |

**结论**: 
- 网络分区期间，多数派 (2/3) 继续服务，强一致性不受影响
- 被隔离节点自动关闭 postgres，防止服务过期数据 → **无脑裂**
- 网络恢复后，数据自动收敛，无需人工干预

### 4.5 级联故障 (Cascading Failure)

| 阶段 | 存活节点数 | 集群状态 | 行为 |
|------|-----------|---------|------|
| 隔离 region1 | 2/3 | ✅ 正常服务 | 读写成功，Raft 多数派存活 |
| 继续隔离 region2 | 1/3 | ❌ 集群下线 | 所有节点 postgres 关闭，无法读写 |
| Heal 全部 | 3/3 | 需手动恢复或等待 | 自动恢复需重启 tserver |

**结论**:
- RF=3 集群可以容忍 **1 个节点故障**（2/3 多数派仍可选举 leader）
- **2 个节点故障 = 集群不可用**，这是 RF=3 的固有限制 (N=3, 容忍 floor((N-1)/2)=1)
- 生产环境建议 RF=5 或跨 5 个 zone 部署以容忍 2 节点故障

## 关键结论

1. **Raft 共识高效**: RF=3 集群停掉 1 个节点后 ~400ms (docker stop) 自动恢复；iptables 网络分区 ~987ms (含 heartbeat timeout)
2. **Read Committed 稳定**: 冲突事务正确排队等待，无数据异常
3. **Geo-Partitioning 灵活**: 支持按 region 分配数据和 Leader Preference
4. **tc netem 延迟模拟有效**: 通过 `NET_DELAY_MS=30` 配合 `iproute-tc` 在单机 Docker 上成功模拟了跨 region 网络延迟（region1=30ms, region2=60ms, region3=90ms），延迟对读写性能的影响呈线性关系，验证了 YugabyteDB 在跨 region 场景下的行为特性
5. **网络分区模拟 (chaosctl)**: 基于 iptables 的混沌控制器可以模拟更真实的网络故障，比简单的 `docker stop` 更贴近生产环境。分区节点自动关闭数据库防脑裂，多数派节点继续提供强一致服务
